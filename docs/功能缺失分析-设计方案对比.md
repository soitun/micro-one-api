# 功能缺失分析 — 设计方案 vs 当前实现

> 基于 `docs/One-API微服务改造方案.md` 与当前代码库的全面对比，分析日期：2026-05-02

## 一、高优先级缺失（核心链路可靠性）

### 1. Redis 集成 — 完全缺失

- **设计方案要求**：配置变更/渠道变更通过 Redis Pub/Sub 实时广播；缓存层使用 Redis
- **当前状态**：7 个服务的 config 都声明了 `RedisConfig`，但 `go.mod` 中连 Redis 客户端库都没有，所有字段都是死代码
- **影响**：渠道变更只能靠轮询感知；限流器是进程内的，多实例无法协同

### 2. 事件总线/消息队列 — 完全缺失

- **设计方案要求**：Phase 3 异步化核心（消费日志、监控统计、告警通知走事件驱动）
- **当前状态**：`events/events.go` 只有 6 个 topic 常量，从未被任何代码引用；无 NATS/Kafka/RabbitMQ 依赖
- **影响**：所有服务间通信都是同步 gRPC，主链路延迟无法优化

### 3. 幂等键机制 — 缺失

- **设计方案要求**：`ReserveQuota(request_id)` 幂等、`CommitQuota(request_id)` 幂等、避免重复扣费
- **当前状态**：`request_id` 由 `fmt.Sprintf("req_%d", time.Now().UnixNano())` 生成，没有去重检查；同一请求重复提交会产生两个独立 reservation
- **影响**：网络重试或客户端重发可能导致重复计费

### 4. 账务对账/审计任务 — 缺失

- **设计方案要求**：每日自动对账用户、Token、渠道消耗；`ledger_entries` 流水可审计
- **当前状态**：仅有 expired reservation 清理 job，无对账逻辑、无审计报告、无异常检测
- **影响**：额度不一致无法自动发现

## 二、中优先级缺失（可观测性与运维）

### 5. 分布式追踪 (OpenTelemetry) — 未实现

- **设计方案要求**：全链路 `trace_id` / `request_id`
- **当前状态**：`internal/pkg/xtrace/` 只有一个 README，零代码
- **影响**：跨服务调用链路无法追踪，故障定位困难

### 6. 健康检查不完整

- **设计方案要求**：所有服务健康检查、就绪检查
- **当前状态**：9 个服务中 4 个缺失（billing、identity、admin、channel）；命名不统一（`/health` vs `/healthz`）
- **影响**：K8s 探针无法正确配置，流量可能打到未就绪实例

### 7. Config-Service 无变更通知

- **设计方案要求**：配置变更发布事件，订阅式本地缓存
- **当前状态**：`SetConfig` 直接写 DB 返回，无任何副作用
- **影响**：配置修改后其他服务无法实时感知

### 8. Channel-Service 无变更通知

- **设计方案要求**：管理端更新渠道后发布 `ChannelChanged` 事件
- **当前状态**：所有 CRUD 操作直接调 repo，无事件发布
- **影响**：渠道状态变更传播依赖轮询周期

## 三、中低优先级缺失（功能完整性）

### 9. OAuth/SSO — 缺失

- **设计方案要求**：identity-service 支持 OAuth、Session/JWT 签发
- **当前状态**：仅有用户名/密码登录，无 OAuth2、OIDC、社交登录；密码存储无 bcrypt（accepts any non-empty password）

### 10. 密码安全 — 未达标

- **当前状态**：`Login()` 接受任意非空密码，无哈希比对
- **影响**：认证形同虚设

### 11. 分组倍率未外部化

- **设计方案要求**：倍率配置可管理
- **当前状态**：`GroupRatio()` 硬编码 `default=1.0, vip=0.5, svip=0.3`；config 中声明了 `GroupRatios map[string]float64` 但未接入

### 12. 用户额度账户模型未升级

- **设计方案要求**：新建 `accounts`、`token_accounts`、`ledger_entries`、`quota_reservations`、`billing_orders` 表
- **当前状态**：`billing_reservations` 和 `billing_ledgers` 已建，但用户额度仍直接读写 `users.quota`/`users.used_quota`，未建立独立账户模型

### 13. Admin API / Identity / Channel 的 HTTP Server 是空壳

- **当前状态**：`admin/server/http.go`、`identity/server/http.go`、`channel/server/http.go` 都是 `func NewHTTPServer() {}` 空实现
- **影响**：这些服务只能通过 gRPC 访问，无法直接 HTTP 健康检查或调试

## 四、低优先级缺失（Phase 4 平台化）

| 缺失项 | 设计方案要求 | 当前状态 |
|--------|-------------|---------|
| 物理分库 | identity_db/channel_db/billing_db/log_db | 共用单 MySQL |
| Service Mesh | K8s + Istio/Linkerd | 仅有基础 K8s manifests |
| 分布式配置版本管理 | `config_revisions` 表 | 无版本追踪 |
| 日志归档清理 | 历史日志定期归档 | 无归档逻辑 |
| 通知模板引擎 | 邮件/站外通知模板 | 仅有 webhook 类型 |

---

## 五、已正确实现的部分

以下设计方案的核心目标**已达成**：

- **服务拆分**：9 个独立服务，单仓多服务架构
- **Relay 无状态化**：零数据库依赖，纯 gRPC 调用下游
- **账务预留模型**：Reserve/Commit/Release 三阶段 + 过期清理
- **重试编排**：指数退避 + 渠道降级
- **模型映射**：22 种上游 provider + 能力追踪
- **统一错误码**：14 个 reason code + HTTP 状态映射
- **部署就绪**：Dockerfile + docker-compose + K8s manifests
- **安全扫描 CI**：gosec + govulncheck + gitleaks + Trivy
