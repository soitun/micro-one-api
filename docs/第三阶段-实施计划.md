# Billing Service 迁移实施计划

## Context

将账务相关功能从单体应用迁移到独立的 `billing-service`，实现配额管理的中心化和一致性保证。这是第三阶段迁移，目标是：
- 建立独立的配额管理中心
- 实现预扣-提交-释放的配额管理机制
- 提供完整的账务审计能力
- 支持用户分组倍率管理

## 实施步骤

### Step 1：创建 Proto 文件

**任务：**
1. 创建 `api/billing/v1/billing.proto`
2. 定义所有消息类型（ReserveQuotaRequest/Response, CommitQuotaRequest/Response 等）
3. 定义 BillingService 服务接口
4. 生成 Go 代码：`make api`

**Proto 定义（关键消息类型）：**

```protobuf
syntax = "proto3";
package billing.v1;
option go_package = "micro-one-api/api/billing/v1;billingv1";

service BillingService {
  rpc ReserveQuota(ReserveQuotaRequest) returns (ReserveQuotaResponse);
  rpc CommitQuota(CommitQuotaRequest) returns (CommitQuotaResponse);
  rpc ReleaseQuota(ReleaseQuotaRequest) returns (ReleaseQuotaResponse);
  rpc GetAccountSnapshot(GetAccountSnapshotRequest) returns (GetAccountSnapshotResponse);
  rpc TopUpQuota(TopUpQuotaRequest) returns (TopUpQuotaResponse);
  rpc CreateRedeemCode(CreateRedeemCodeRequest) returns (CreateRedeemCodeResponse);
  rpc RedeemCode(RedeemCodeRequest) returns (RedeemCodeResponse);
  rpc ListLedger(ListLedgerRequest) returns (ListLedgerResponse);
}
```

### Step 2：创建数据库表

**任务：**
1. 创建 `migrations/billing/` 目录
2. 创建以下迁移脚本：
   - `001_create_billing_reservations.sql`
   - `002_create_billing_ledgers.sql`
   - `003_create_billing_redeem_codes.sql`
   - `004_create_billing_redeem_records.sql`

**关键表结构：**

```sql
-- billing_reservations 表
CREATE TABLE `billing_reservations` (
  `id` bigint unsigned NOT NULL AUTO_INCREMENT,
  `reservation_id` varchar(64) NOT NULL COMMENT '预扣 ID',
  `user_id` varchar(64) NOT NULL COMMENT '用户 ID',
  `request_id` varchar(64) NOT NULL COMMENT '请求 ID',
  `amount` bigint NOT NULL COMMENT '预扣金额',
  `status` varchar(20) NOT NULL COMMENT '状态：reserved/committed/released/expired',
  `model` varchar(64) DEFAULT NULL,
  `channel_id` varchar(64) DEFAULT NULL,
  `created_at` datetime(3) DEFAULT NULL,
  `updated_at` datetime(3) DEFAULT NULL,
  `expired_at` datetime(3) DEFAULT NULL COMMENT '过期时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `idx_reservation_id` (`reservation_id`),
  KEY `idx_user_id` (`user_id`),
  KEY `idx_request_id` (`request_id`),
  KEY `idx_status_expired` (`status`, `expired_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- billing_ledgers 表
CREATE TABLE `billing_ledgers` (
  `id` bigint unsigned NOT NULL AUTO_INCREMENT,
  `user_id` varchar(64) NOT NULL,
  `amount` bigint NOT NULL,
  `balance_after` bigint NOT NULL,
  `type` varchar(32) NOT NULL COMMENT 'consume/recharge/refund/redeem',
  `reference_id` varchar(64) DEFAULT NULL,
  `remark` text,
  `created_at` datetime(3) DEFAULT NULL,
  PRIMARY KEY (`id`),
  KEY `idx_user_id` (`user_id`),
  KEY `idx_type` (`type`),
  KEY `idx_created_at` (`created_at`),
  KEY `idx_reference_id` (`reference_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 关键难点和解决方案

### 1. 并发控制

**问题：** 高并发场景下，多个请求同时预扣配额可能导致超支

**解决方案：**
- 使用数据库乐观锁（版本号）或悲观锁（SELECT FOR UPDATE）
- 使用 Redis 分布式锁
- 配额扣减操作使用原子操作

### 2. 数据一致性

**问题：** 预扣和提交之间数据不一致

**解决方案：**
- 使用数据库事务保证原子性
- 实现幂等性设计（reservation_id 唯一）
- 添加定时对账任务
- 预扣过期自动释放

### 3. 分布式事务

**问题：** 跨服务调用导致事务失败

**解决方案：**
- 实现重试机制（指数退避）
- 实现补偿机制（释放预扣）
- 添加监控告警

### 4. 性能优化

**问题：** 高并发场景下性能不足

**解决方案：**
- 使用 Redis 缓存账户信息
- 数据库连接池优化
- 批量更新优化
- 异步写入流水记录
