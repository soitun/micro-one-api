## 2. P0 — 阻塞生产可用性

### 2.1 docker-compose 补全

**现状**: `deployments/docker-compose/docker-compose.yml` 仅包含 relay-gateway / identity-service / channel-service。

**补充内容**:

- 添加 `billing-service` 服务定义（端口 8004/9004）
- 添加 `admin-api` 服务定义（端口 8000/9000）
- 统一所有服务使用 Dockerfile 构建（当前 identity/channel 使用 `go run`）
- 添加 MySQL / Redis 依赖服务
- 添加 `depends_on` 保证启动顺序

**涉及文件**:
- `deployments/docker-compose/docker-compose.yml`

**验收标准**: `docker-compose up` 能启动全部 5 个服务 + MySQL + Redis，各服务健康检查通过。

---

### 2.2 relay 重试策略

**现状**: `internal/relay/` 下无任何重试相关代码。上游模型调用失败后直接返回错误。

**补充内容**:

- 在 `internal/relay/biz/` 下新增 `retry.go`
- 实现指数退避重试策略（可配置最大重试次数、基础间隔、最大间隔）
- 在 provider 调用层集成重试逻辑
- 支持按错误类型决定是否重试（如 429 限流重试，400 参数错误不重试）

**涉及文件**:
- `internal/relay/biz/retry.go`（新增）
- `internal/relay/provider/factory.go` 或 `provider.go`（集成重试）
- `configs/relay-gateway.yaml`（添加重试配置项）

**验收标准**: 上游返回 429/5xx 时自动重试，400/401 等客户端错误不重试，重试次数和间隔可配置。

---

### 2.3 模型映射配置

**现状**: 无 models.yaml 或任何模型映射配置。无法支持模型别名（如 `gpt-4o` → 具体渠道模型名）和能力路由。

**补充内容**:

- 创建 `configs/models.yaml` 定义模型能力矩阵
- 在 `internal/relay/biz/` 下新增 `model_mapping.go`
- 实现模型别名解析（客户端模型名 → 实际渠道模型名）
- 实现能力查询（模型是否支持 function_call / vision / streaming 等）
- 在 channel-service 的 `SelectChannel` 流程中集成模型映射

**涉及文件**:
- `configs/models.yaml`（新增）
- `internal/relay/biz/model_mapping.go`（新增）
- `internal/channel/biz/` 或 `internal/relay/biz/route.go`（集成）

**验收标准**: 客户端请求 `gpt-4o` 能正确映射到渠道实际模型名，模型能力查询返回准确的能力矩阵。

## 3. P1 — 提升可维护性

### 3.1 wire.go 依赖注入源文件

**现状**: 5 个服务均有 `wire_gen.go`（手写或生成），但无对应的 `wire.go` 源文件。无法通过 `wire ./cmd/xxx` 重新生成。

**补充内容**:

- 为每个服务创建 `wire.go`，定义 `New*Set` 注入器函数
- 确保 `wire.go` 中的 ProviderSet 与 `wire_gen.go` 中的实际注入逻辑一致
- 验证 `wire ./cmd/xxx` 能正确生成 `wire_gen.go`

**涉及文件**（每个服务一个）:
- `cmd/relay-gateway/wire.go`
- `cmd/admin-api/wire.go`
- `cmd/identity-service/wire.go`
- `cmd/channel-service/wire.go`
- `cmd/billing-service/wire.go`

**验收标准**: 每个服务的 `wire.go` 定义 Injector 函数，`wire` 命令能重新生成 `wire_gen.go` 且结果与当前一致。

---

### 3.2 K8s 部署补全

**现状**: `deployments/k8s/deployment.yaml` 仅覆盖 relay-gateway。其他 4 个服务无 K8s 资源。

**补充内容**:

- 为 identity-service / channel-service / billing-service / admin-api 各创建独立的 K8s 资源
- 每个服务包含：Deployment + Service + ConfigMap + HPA
- 拆分 deployment.yaml 为多文件（按服务）
- 添加 Ingress 资源（relay-gateway 对外，admin-api 管理端对外）

**涉及文件**:
- `deployments/k8s/identity-service.yaml`（新增）
- `deployments/k8s/channel-service.yaml`（新增）
- `deployments/k8s/billing-service.yaml`（新增）
- `deployments/k8s/admin-api.yaml`（新增）
- `deployments/k8s/ingress.yaml`（新增）

**验收标准**: `kubectl apply -f deployments/k8s/` 能部署全部 5 个服务，各服务可通过 ClusterIP 互通。

## 4. P2 — 功能完善

### 4.1 admin-api 日志查询

**现状**: admin-api 无日志查询功能。

**补充内容**:

- 在 `admin.proto` 中添加 `ListLogs` / `GetLog` RPC（或复用 billing 的 ListLedger 思路）
- 在 admin-api 中实现日志查询服务端
- 日志数据源可先走数据库查询（后续接入 log-service）

**涉及文件**:
- `api/admin/v1/admin.proto`（新增 RPC）
- `internal/admin/service/admin.go`（实现）

---

### 4.2 UpdateSystemOptions 持久化

**现状**: `internal/admin/service/admin.go` 中 `UpdateSystemOptions` 返回 `"not implemented"`。

**补充内容**:

- 定义系统配置的数据库表结构
- 在 `internal/admin/data/` 中实现系统配置的 CRUD repo
- 在 `internal/admin/service/admin.go` 中对接 repo 完成持久化

**涉及文件**:
- `migrations/` 下新增系统配置表迁移
- `internal/admin/data/` 下新增系统配置 repo
- `internal/admin/service/admin.go`（对接 repo）

---

### 4.3 relay-gateway proto 补全

**现状**: `api/relay/v1/relay.proto` 中 `service RelayService {}` 为空。

**说明**: relay-gateway 当前使用 HTTP 直接对外，不依赖 gRPC 契约。此项为架构完整性补齐，非阻塞项。

**补充内容**:

- 在 `relay.proto` 中定义内部 RPC（如 `RelayChatCompletion` / `RelayEmbedding` 等）
- 用于未来支持 gRPC 内部调用或服务间 relay

**涉及文件**:
- `api/relay/v1/relay.proto`
